## 怎么跑
- 本机开发:`python webui/run_dev.py` → http://localhost:8601(改前端 no-store 即时生效;改 .py 需重启;已内置带时间戳日志)。
- 生产:`uvicorn webui.api_server:app --port 8601 --log-config webui/log_config.json`(走 .env,USE_POSTGRES=true → 真 PG)。`--log-config` 给 uvicorn 日志加时间戳(默认不带)。
- 定时任务:supervisor 常驻 `register_default_jobs(); serve_forever()`(生产服务器目录,独立 venv;**venv 别从别处复制**,见末节坑)。
- MCP(Agent 接入,80+ 工具):`python mcp_server.py`。

## 关键约定(动代码前必读)

## ✅ 已破案(2026-06-30):"全源同时挂"根因 = `data/rate_limiter.py` 全局锁

**真凶**:`data/rate_limiter.py` 在 2026-06-30 之前用了**全局唯一一把锁** + 锁里 `time.sleep(wait)`。一个 tickflow 调用持锁 sleep 6s 期间, 妙想/腾讯/新浪/东财/问财... 全部源的并发调用都被挡在 `with _lock:` 外面排队;strategy_prefetch 16 worker + datahub-route 32+ worker 抢这把锁, 任务被串行化成 90s+ 假超时, 看似"全源同时挂"。

**定位过程**:埋点显示"函数已跑但没完成 + throttle 阻塞 0 次" → 走错了 4 条假说(机房 TLS 重置 / 孤儿线程 / socket 泄漏 / urllib bug)→ 最后 `sudo venv2/bin/py-spy dump --pid <webui-pid>` 一秒实证:几乎所有线程都堵在 `rate_limiter.py:58 with _lock:`。 详见 memory [[rate-limiter-per-key-lock-fix]]。

**修法**:per-source 独立锁(`_lock_for(source)`),与 `core/rate_limiter.py` 对齐。同源仍按 `_gap` 串行(原意),不同源互不阻塞。一次 commit:`data/rate_limiter.py`。

**实战验证(修复当晚 12:18 跑通)**:
| 任务 | 修前 | 修后 |
|---|---|---|
| strategy_prefetch 11 策略 | 631s 全失败 | **40s 全成功** |
| 妙想 5 策略 | 5×90s 全 timeout | **各 1-3s** |
| 主力资金 | 90s timeout | 26s ✅ |
| unified_selection | 1800s 强切断 / 0 入池 | **~19min / 6 只入池 + 推送** |

修后仍能看到 baostock K线 10s 单源超时、东财板块 RemoteDisconnected、问财熔断等 — 这些都是**单源临时挂**,跳源/熔断机制工作正常,**再没出现"全源同时挂"的雪崩**。

**遗留诊断埋点(可保留观测,不急撤)**:
- `data/datahub.py::_route` 拆"池排队 vs 函数慢"(env `DATAHUB_ROUTE_DEBUG=0` 可关)
- `core/rate_limiter::throttle` 慢日志 ≥3s(env `RATE_LIMIT_SLOW_LOG=0` 可关)

**新会话排查类似卡死必杀技**:`sudo venv2/bin/py-spy dump --pid <pid>` — 看 Python 堆栈直接定位被卡的代码行;比 strace/tcpdump 快 10 倍。命令行能跑 + 长跑进程跑不通,几乎都是并发/状态污染(锁/连接池/缓存),不是代码逻辑本身。
