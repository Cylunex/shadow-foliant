## 怎么跑
- 本机开发:`python webui/run_dev.py` → http://localhost:8601(改前端 no-store 即时生效;改 .py 需重启;已内置带时间戳日志)。
- 生产:`uvicorn webui.api_server:app --port 8601 --log-config webui/log_config.json`(走 .env,USE_POSTGRES=true → 真 PG)。`--log-config` 给 uvicorn 日志加时间戳(默认不带)。
- 定时任务:supervisor 常驻 `register_default_jobs(); serve_forever()`(生产服务器目录,独立 venv;**venv 别从别处复制**,见末节坑)。
- MCP(Agent 接入,80+ 工具):`python mcp_server.py`。

## 关键约定(动代码前必读)
1. **统一数据层 `data/datahub.py` 是取外部数据的唯一门面**。一律 `import datahub` 调用(quotes/kline/indices/north_flow/dragon_tiger/sector/financials/valuation/news/screen/convertible_bonds/stock_reports/industry_reports/lockup_expiry…),**别直接 import a_stock_data_adapter / data_source_manager / 拼外部域名**(那些是 datahub 底下的内部源)。加新源 = 往对应域 `_route(...)` 加一项。
2. **datahub 三级缓存**:L0 进程内存 → L1 Redis(JSON友好值)→ L2 本地文件 pickle(含 DataFrame)。每域按时效配 TTL;`use_cache=False` 或 env `DATAHUB_CACHE=false` 绕过;`datahub.cache_stats()`/`cache_clear()` 观测清理。K线另有专用磁盘缓存 `db/kline_cache`(盘后 `kline_prefetch` 焐热;`StockDataFetcher` 也共享此缓存),**分 raw / qfq 两套**(见关键约定 7「K线复权口径」)。
3. **jobs 唯一调度入口 `jobs_hub._wrap`**:REGISTRY 内任务按 `automation_config` 开关闸门控;**任务抛异常 → job_runs 记 error(带 traceback)+ 推告警**。每日市场/分析任务都有 `_skip_if_not_trading` 守卫;周末任务(weekly_analysis 周日、mx_weekend_outlook 周日、portfolio_stress_ai 周日、ai_eval_weekly 周一)**不守卫**(周末/假日照跑)。
   - ⚠️ **任务节奏 2026-06-25 大改**(以 `register_default_jobs` 实际代码为准;本文下方个别旧时点如 14:40/16:02/15:48 已过时):**监控降噪**(stock_monitor_check 进场区间盯盘退役、急跌并入 11:20 noon_portfolio;ai_rec_check 推荐池胜率回填由盘中 every:30 改**盘后 16:35 收盘价后验**);**持仓三点**(09:50 morning 全持仓挑 top15 候选存 focus_candidates → 11:20 noon 只看候选 → 14:30 afternoon 尾盘总结);**盘后全挪 16:30 后**(16:30 kline_prefetch 链头焐缓存 → factor/快照/decision_outcomes → 18:30 龙虎榜 → 18:35 公告 → 19:00 daily_backtest;依赖硬约束:kline_prefetch 最先)。死注册(selection_debate/lockup_radar/research_digest)已删。详见 memory [[task-flow-consolidation]]。**盘前预取主力选股(2026-06-26)**:新增 09:15 `main_force_prefetch` 预热问财「主力资金净流入排名」入当日缓存 → 09:45 `unified_selection` 的主力资金策略读缓存(`main_force_selector.get_main_force_stocks_cached`),问财熔断时主力选股不再退化成「按市值选股」;预取失败不告警(09:45 自动 fallback 现调)。
4. **盈利闭环**:选股(unified_selection / wf_daily_strategy_scan)→ 推荐池(save_recommendation,记 ref_price)→ 盘中追价(ai_rec_check 回填 last_price/realized;了结按委托价 tp/sl 口径)→ 周评估(ai_eval_weekly 真实胜率,零收益不计胜负分母)→ 回喂门槛(_source_feedback,conservative 收紧)。**并行**:所有分析/选股/AI 能力的结论都写 `decision_signal`,16:10 按 K线方向后验,形成可量化的第二闭环。
5. **任务默认全开**:automation_config 内任务 default 现全 True。生产开关以 DB `automation_switches` 为准(无记录才回退 default)。
6. **部署铁律(2026-06)**:Python module import 后缓存,**改代码必须重启进程才生效**(supervisor autorestart 只在进程退出时触发,不感知代码变更)。一键部署 `bash scripts/deploy.sh`(= git pull + supervisorctl restart stock-jobs-hub stock-webui)。历史踩坑:git pull 了没 restart → 跑的还是老代码,反复"修了没用"。
7. **数据层韧性(2026-06,全在 `data/datahub._route`)**:
   - **每源硬超时**:`_route` 每个源调用套独立线程池 + `future.result(timeout)`,默认 `_SOURCE_TIMEOUT=20s`(env 可调)、kline 45s。源卡死必在 timeout 内切下一个源,**绝不无限等**(根治"外部接口卡死拖垮任务")。
   - **健康度评分** `_health`:成功率为主 - 延迟轻罚,连续失败进冷却(-1.0 沉底);**未知源给 0.5(中性,不是 1.0)** —— 否则新加的源会盖过"已验证健康"源(rate1.0 但有延迟惩罚=0.97)抢主位,正是 2026-06 把东财放 kline 主源、海量调用打到 IP 被封的根因。现在:已验证好用的源稳居主位、新源仅主源失败时才上、源链静态顺序在同分时(stable sort)生效 → **可达源列前真正起作用**。
   - **限流** `data/rate_limiter`:按源最小间隔,**只增不减**(`_gap_for` 以 1s 惯例为下限钳制):akshare 3s / pywencai·同花顺 2s / 其余(批量行情等)1s。
   - **K线复权口径(2026-06-24 改为两套缓存)**:`datahub.kline(code, period, adjust='raw'|'qfq')`。
     - **raw 不复权**(默认,缓存键无后缀):回测/持仓盈亏/决策后验/显示真实价。源:新浪`scale=240&ma=no`→东财`fqt=0`→通达信。
     - **qfq 前复权**(缓存键 `_qfq` 后缀):InStock/形态/缠论/因子/技术指标/筹码(除权跳空毁形态,行业标准)。源:东财`fqt=1`→akshare qfq;取不到 fallback raw 但**不写 qfq 缓存**防污染。新浪/通达信只入 raw。
     - 两套缓存键隔离、互不污染;`prefetch_kline(adjust='both')` 盘后焐两套(qfq 仅东财快速源,封时跳过)。成交量统一"股"(东财/akshare 手×100、通达信自适应)。详见 memory `kline-adjust-two-cache`。
     - ⚠️ 切因子库口径后 **`kline_daily` 表须 TRUNCATE 重采**(否则历史 raw + 新 qfq 混)。改任一 K线源复权/单位前先核对共享缓存。
   - **全源熔断(2026-06-25,根治雪崩)**:`_route` 开头若该域**所有源都在活跃冷却期**(连续失败 `_health<-0.5`)→ 直接返回 empty,**不再逐源吃满超时**。根治"外网全挂时 监控/选股/快照 逐只吃 quotes60s+kline135s → 把 `_ROUTE_POOL`(64线程)占满 → 后续 submit 排队也算假超时 → 级联雪崩"。冷却 120s 后自动放行重试、网络恢复即自愈。
   - **慢源盘后预热(2026-06-25,雪崩另一半)**:⚠️**盘中绝不逐只现调慢源**。`full_valuation`(同花顺一致预期,走 `_route` 全源熔断)、`pywencai`(问财,限流2s+30s超时,**不走 `_route`,故 `pywencai_safe` 自带熔断**:连续失败2次→冷却120s内直接短路降级)盘中 60 只逐只现调会池雪崩。改:`collect_factors`/`full_valuation` 整体缓存**1天**(基本面/一致预期几天才变)+ `kline_prefetch`(15:35盘后)顺便焐 `collect_factors`(内含 full_valuation+pywencai)→ 盘中读暖缓存 0 调慢源。新增盘中逐只批量取数务必走缓存域或盘后预热。详见 memory `datahub-multisource-status`。
     - **2026-06-25 补**:问财熔断是补上的最后一个洞 —— 此前问财不可达时 `kline_prefetch` 焐 391 只每只吃满30s,拖到 1800s 硬超时切断 + 误报「⚠️任务异常: kline_prefetch」(K线明明 2 分钟已全焐完);`factor_collection`(1200s)同病。现两慢源都连续失败后秒级短路,焐热/采集任务网络全挂时也几分钟内干净收尾、不再误报。
     - **2026-06-26 补(主力选股问财预取)**:`collect_factors` 预热的是**逐只因子**的问财查询,而**主力选股是另一条"全市场主力资金净流入排名"问财查询**,原在 09:45 选股高峰现调。新增**盘前** `main_force_prefetch`(09:15)预取入当日缓存(`db/main_force_cache`,键=日期+days_ago+市值档,跨交易日自然失效)→ 09:45 读缓存 0 调问财。⚠️ **任何盘中现调问财/慢外部服务的逐只或全市场查询,都应盘前/盘后预取入当日缓存**(下一个候选:妙想 `miaoxiang.query` 零缓存,`mx_selection_review` 10:30 逐只串行诊断 10 只)。
8. **全局任务硬超时(`jobs_hub._run_with_log`)**:每个定时任务套外层 `future.result(timeout)`,默认 10min,慢任务在 `_TASK_HARD_TIMEOUTS` 单独配(rag_ingest/daily_backtest 1h、unified_selection 20min…)。超时 → 切断 + 发**平和提示** `📡 数据源暂不可用`(report 类,非吓人的 `⚠️任务失败`+traceback)。真代码异常才发 alert 告警。⚠️ **`_log_run(status='error')` 本身会无条件自动推一条 `⚠️任务异常` alert**(供任务内部自处理错误用),故 `_run_with_log` 的超时/异常分支传 **`notify=False`**——否则同一次失败叠发"吓人告警 + 平和提示/带traceback告警"双推送(2026-06-26 修;曾让 `factor_collection` 超时同时收到 `⚠️任务异常` + `📡数据源暂不可用`)。
9. **统一日志(`core/log_setup`,_bootstrap 自动初始化)**:所有裸 `print` 自动加 `YYYY-MM-DD HH:MM:SS` 时间戳(行首已有 ts / uvicorn 风格不二次加);过滤 akshare/PyExecJS 残留的整页 HTML 噪音 + Node punycode 警告(`NODE_OPTIONS=--no-deprecation`)。新代码可 `from core.log_setup import get_logger`。开关 `SHADOW_LOG_TIMESTAMPS=false` / `SHADOW_LOG_FILTER_HTML=false`。
10. **通知分类路由(`notify/notification_router`)**:`send(category, title, content)` 唯一入口。category=alert/report/archive/system_error/daily_summary,每类 env `NOTIFICATION_ROUTE_<CAT>=qq,email,...` 配渠道;**title 关键字路由** `NOTIFICATION_ROUTE_REPORT__KW_<词>=...`(title 含「词」时优先)。⚠️ **qq 与 dingtalk 是两个独立渠道**(本项目 dingtalk 协议实际转投 QQ,故二者都≈QQ;email 才是邮件)。默认不发隐式 email 兜底(严格按路由)。每条推送打审计日志 `[notify] <cat> "title" → ch=ok`。红涨绿跌(涨/盈🔴 跌/亏🟢)。AI 推荐池(ai_recommendations)只后台跟踪胜率**不推通知**,只有真持仓(monitored_stocks)才推止盈止损。

## 关键坑与教训(动这些代码前看)
- **`get_indicator_snapshot(key)` 直接返回 indicators 列的反序列化结果,没有 'indicators' 外层包裹**:`save_indicator_snapshot(k, {'picks':[...]})` 存什么读出来就是什么。历史上多处误写 `snap.get('indicators')` → 恒 None → 死分支(`_last_selection` 的盘后扫描并入 / 妙想复核 / mx_advisor 全踩过)。读 `_last_selection` 一律走 `jobs_hub._last_selection_picks()`,别裸读。
- **盘后扫描买入判定别用 `'buy' in text` 子串**:plan_execute 自由文本里"不建议买/avoid/选项回显"会被误判为买入、污染推荐池与胜率。已改中文正向词 + 否定优先门;`decision_signal.normalize_action` 也是否定优先。
- **批量解析 LLM 逐行结论用 `re.search` 不用 `re.match`**:LLM 偶尔给 `- `/`1.`/`·` 前缀,行首锚定会整批静默丢空。
- **逐只外部抓取的批量函数**三大坑:①串行 ②用无缓存源冷拉卡死 ③隐藏 N² 全持仓遍历。接入必做:走 `datahub` 缓存源、并发、超时护栏、按市值 top-N、部分失败不缓存、批量取价(`datahub.quotes` 一次拉)。
- **盘中逐只现调慢源会触发线程池雪崩(2026-06-25 实战踩坑)**:`取因子`/选股/监控盘中逐只调 `full_valuation`(同花顺)、`pywencai`(问财,`pywencai_safe` 自带熔断,2026-06-25)、`kline(qfq)`(东财),每只吃满 45s/30s 超时 → 把 `_ROUTE_POOL`(64线程)占满 → 后续 submit 排队也算假超时 → **看似"全源超时(网络挂)"实为池耗尽级联雪崩**(铁证:同时段 `quotes` 批量83只仍 4.4s)。已修:`_route` 全源熔断 + `collect_factors`/`full_valuation` 1天缓存盘后焐 + 加仓审核 `limit15`+`with_fundamental=False` + `stock_monitor_check` 尾接子任务套整体超时。**新写盘中批量逐只取数,慢域一律走缓存/盘后预热,别现调**。
- **`webui/_jsonsafe` 已兜底 numpy 标量**:`np.bool_`/`np.float64` 不是 JSON 原生类型,会让 FastAPI 序列化抛 500;返回含 numpy 值的端点靠它递归 `.item()` 降级。
- **写 PG JSONB 列前必须洗 NaN(2026-06-26)**:`json.dumps` 默认 `allow_nan=True`,把 `float('nan')`/`Inf` 原样写成字面量 `NaN`/`Infinity` → **PG 的 json/jsonb 列拒收**(`invalid input syntax for type json: Token "NaN" is invalid`),SQLite 不校验所以本地发现不了。`save_market_snapshot`/`save_indicator_snapshot` 写库前过 `jobs_hub._json_sanitize`(递归 NaN/Inf→null,np.float64 也通吃)。datahub 行情/北向记录常含 NaN(如 turnover_ratio) → `daily_market_snapshot` 曾因此 0.9s 报错。**新增任何 `json.dumps(...)` 写 JSONB 都要先洗。**
- **多源聚合端点按源分缓存 + 拒空**:否则某源瞬时失败会被连同好数据一起缓存。
- **了结/胜率口径**:推荐到期了结必置 `is_active=0`(否则 list_active 反复捞回);止盈止损 realized 用委托价 tp/sl(非快照价,跳空会失真);评估零收益样本不计胜负分母。
- **LLM 必须有超时**:`llm_router.call`/`deepseek_client.call_api` 已加 per-call timeout(`LLM_TIMEOUT` 默认40s,thinking 120s)+ `max_retries=0`。LLM 是规则之上的增强层,挂起 provider 不得阻塞主路径。
- **venv 不可复制**:跨项目/机器复制必坏。一项目一 venv、`python -m venv` 现建、Python 小版本一致。
- **mootdx 已移出主依赖**(httpx pin 与 mcp 冲突):可选 K线兜底源,需要时单独装 `pip install "mootdx>=0.11.0" && pip install -U "httpx>=0.27.1"`。
- **日志时间戳**:已由 `core/log_setup`(_bootstrap 自动装)统一给所有 print 加时间戳 + 过滤 HTML/Node 噪音,不再需要 supervisor `| ts`。uvicorn 仍用 `--log-config webui/log_config.json`。
- **外部源单点故障(2026-06-24 大幅补源)**:核心域已做 datahub 级**真·跨公司多源**——`quotes` 腾讯+东财ulist+新浪、`kline` 新浪+东财+通达信(mootdx,需 pip install,真独立协议源)、`sector_spot` 新浪行业+同花顺行业。**2026-06-25 去东财依赖**:`sector_ranking`/`sector_fund_flow`(原全东财)加同花顺 `data.10jqka`(akshare ths 纯 JSON 真跨源)、`kline_qfq`(原 east+akshare 全东财)加 `sina_qfq`(新浪 `stock_zh_a_daily`,唯一真非东财 qfq);`pywencai` 加自带熔断。`capital_flow`/`stock_news` 加了 akshare 兜底但**实测同走东财**(弱兜底,东财封时同死);`north_flow` 统一走 `datahub.north_flow`(单位亿元)。加源时格式必须与既有源**逐字段一致**(kline = DatetimeIndex 'Date' + 大写 OCHLV + 成交量"股"),否则 177 处调用方踩雷;新源用 `scripts/smoke_test_datahub_sources.py` 核对字段同构。详见 memory `datahub-multisource-status`。
- **`datahub.quotes` 归一 key 成6位,传带前缀/指数代码会取空(2026-06-26)**:`quotes()` 返回前用 `_norm_code` 把 key 归一成6位(`sh000300`→`000300`);调用方若传带前缀/指数代码再按**原 key** `.get` 必取空。**大盘指数走专用 `datahub.indices()`**(新浪/腾讯多源,按中文名匹配),**别用 `quotes()`**(它按个股口径取数,且 `sh000001` 与平安银行 `000001` 重码)。个股直接 `.get(code)` 的调用方加 `or quotes.get(str(code)[-6:])` 兜底(对6位码空操作、对前缀码自动剥前缀);遍历 `.items()` 的调用方天然安全。曾让 daily_signal 盘前/午盘报告的"大盘指数"整段取空。
- **应用层取数已彻底收口 datahub(2026-06-26)**:`jobs`/`scripts`/`agents`/`core`/`webui`/`mcp` 不再直 import `a_stock_data_adapter`/`data_source_manager`(原 factor_collector 裸调 `adapter.get_full_valuation` 逐只无熔断,正是 `factor_collection` 撞 1200s 超时的真凶)。新增两个 datahub 薄封装:`capital_flow_adata`(adata 资金流补充源,形状零变)、`dragon_tiger_detail`(东财 RPT 龙虎榜明细,保原字段口径)。残留直 import 仅 `data/` 内部源模块(`market_sentiment_data` 因走 tushare `daily_basic`/`index_daily`、datahub 无对应方法,暂留;无 tushare key 时本就降级)+ `smoke_test`(合法)。
- **新源别放主源位**:一个新数据源(尤其反爬严的东财)放源链第一位 + 健康度未知给高分 → 会被海量调用打到 IP 封禁。新源进 `_route` 默认排在已验证源**之后**(靠 _health 未知=0.5 + 静态顺序保证)。
